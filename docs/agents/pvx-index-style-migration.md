# PVX 全站风格统一改造流程

## 1. 文档目的

本文用于指导 PVX 项目逐步完成全站视觉风格统一，同时满足以下要求：

- 当前阶段只改造 PVX 首页、捏脸数据、体型数据、家园蓝图四个模块。
- 未进入改造范围的页面，在其单独优化前不得出现布局、样式或交互变化。
- 改造期间避免复制多套近似样式，公共能力应从一开始按可复用方式设计。
- 全部模块改造完成后，可以平滑删除兼容层并沉淀公共设计系统。
- 优先使用项目已全局引入的 Tailwind CSS；LESS 用于设计令牌、复杂状态、第三方组件覆盖和业务差异。
- 改造不得改变现有业务逻辑、接口参数、权限规则、支付下载流程和路由行为。

## 2. 目标风格

统一改造后的页面应具备以下视觉特征：

- 页面使用浅灰蓝背景 `#f6f8fa`。
- 内容使用白色卡片、浅色边框和克制的阴影。
- 主文字使用 Slate 深灰色系，次要文字降低对比度。
- 靛蓝或紫色作为主要强调色，黄色、玫红用于少量状态提示。
- 大区块使用明显圆角，内部控件使用中小圆角。
- 内容通过卡片分组，而不是依赖大面积高饱和背景。
- 标题建议使用“图标块 + 主标题 + 副标题”的层级结构。
- Hover 动效以轻微上移、图标放大和阴影增强为主。
- PC 端保持清晰的信息分栏，移动端收敛为单列布局。

改造时不得新增无作用域的 `body`、`.m-main`、`.u-title`、`.u-icon`、`.u-btn` 等通用选择器样式。

## 5. 样式架构

### 5.1 公共设计基础层

公共设计基础层应在第一次改造时建立，推荐目录：

```text
src/assets/css/design-system/
├── _tokens.less
├── _mixins.less
├── components.less
└── utilities.less
```

职责：

- `_tokens.less`：颜色、圆角、阴影、字号、间距、动画时间和内容宽度。
- `_mixins.less`：卡片、标题、交互态等可复用 LESS mixin。
- `components.less`：仅输出带明确设计系统前缀的样式。
- `utilities.less`：只补充 Tailwind 不适合表达或项目缺失的少量工具类。

`_tokens.less` 和 `_mixins.less` 不得直接输出影响页面的全局 CSS。

推荐的初始令牌：

```less
@pvx-page-bg: #f6f8fa;
@pvx-surface: #ffffff;
@pvx-surface-muted: #f8fafc;

@pvx-text-primary: #0f172a;
@pvx-text-regular: #334155;
@pvx-text-secondary: #64748b;
@pvx-text-muted: #94a3b8;

@pvx-border: #e2e8f0;
@pvx-border-light: #f1f5f9;

@pvx-primary: #5b5cf5;
@pvx-primary-dark: #0f172a;
@pvx-accent-yellow: #f59e0b;
@pvx-accent-rose: #fb7185;

@pvx-radius-page: 48px;
@pvx-radius-card: 24px;
@pvx-radius-control: 16px;
@pvx-radius-small: 10px;

@pvx-shadow-xs: 0 1px 2px rgba(15, 23, 42, 0.05);
@pvx-shadow-hover: 0 12px 24px rgba(15, 23, 42, 0.1);
```

间距统一使用以下序列：

```text
8 / 12 / 16 / 20 / 24 / 32 / 40 / 48
```

### 5.2 公共视觉组件层

推荐从首个改造模块开始建立无业务含义的公共组件：

```text
src/components/design/
├── PvxPageShell.vue
├── PvxSurface.vue
├── PvxSectionHeader.vue
├── PvxToolbar.vue
├── PvxActionButton.vue
└── PvxEmptyState.vue
```

公共组件应满足：

- 通过 slot 承载业务内容。
- 只提供有限且明确的 props/variant。
- 默认值不得改变旧页面行为。
- 优先使用 Tailwind 表达布局、间距和基础外观。
- 不绑定捏脸、体型、家园等具体业务名称。
- 不为了少量差异堆积大量布尔型 props。

### 5.3 模块适配层

推荐目录：

```text
src/assets/css/modules/
├── portal-theme.less
├── face-body-theme.less
└── homeland-theme.less
```

模块适配层只处理：

- 捏脸与体型不同的图片比例和卡片高度。
- 家园地图坐标、标记和工具提示。
- 详情页购买区域等业务视觉。
- 第三方解析器组件样式。
- Element Plus 内部样式覆盖。
- 无法通过静态 Tailwind 类可靠表达的动态状态。

模块样式必须从明确的页面根节点开始，不得输出裸公共选择器。

## 6. Tailwind 与 LESS 使用规范

### 6.1 Tailwind 当前状态

项目使用 Tailwind CSS v4，主要业务入口已全局引入：

```js
import "@/assets/css/tailwind.css";
```

Tailwind Preflight 已关闭，因此不会通过全局 Reset 改变现有页面。

### 6.2 Tailwind 负责的内容

- Flex、Grid 和定位布局。
- Padding、Margin、Gap。
- 宽高、最大宽度和溢出处理。
- 背景、边框、圆角和阴影。
- 字号、字重和文字颜色。
- 常规 Hover、Focus、Transition。
- 常规响应式布局。

示例：

```html
<div class="grid grid-cols-3 gap-6 max-xl:grid-cols-2 max-md:grid-cols-1">
    <slot />
</div>
```

### 6.3 LESS 负责的内容

- 设计令牌和 mixin。
- Element Plus 深层选择器覆盖。
- 第三方组件样式修正。
- 复杂动画和多层状态。
- 业务主题色和 CSS 自定义属性。
- 地图、轮播、解析器等特殊布局。

### 6.4 禁止事项

- 不动态拼接完整 Tailwind 类名，例如 ``bg-${color}-500``。
- 动态类必须使用可被构建扫描到的显式映射。
- 不大量使用 `!important` 或 Tailwind `!` 修饰符压制旧样式。
- 不在模板中堆积难以阅读的超长工具类；重复组合应抽成组件。
- 不为每个模块复制同一组 Tailwind 类。

## 7. 作用域隔离规则

### 7.1 目标模块根节点

四个模块应具备稳定且唯一的主题根类，例如：

```text
.p-pvx-portal
.p-pvx-face
.p-pvx-body
.p-pvx-homeland-root
```

新主题样式必须限定在这些根类或公共设计组件的专属前缀下。

### 7.2 禁止新增的裸选择器

不得在新主题文件中直接编写：

```less
body {}
.m-main {}
.m-header {}
.m-navigation {}
.u-title {}
.u-icon {}
.u-btn {}
.m-nav-item {}
.type-item {}
.el-tabs {}
.el-table {}
```

如确需覆盖，必须增加目标页面根类：

```less
.p-pvx-homeland-root {
    .el-table {
        // 仅家园模块生效
    }
}
```

### 7.3 公共组件迁移规则

已被旧页面使用的公共组件，不得直接改变默认外观。应增加向后兼容的 variant：

```vue
<PvxSearch variant="modern" />
```

```js
props: {
    variant: {
        type: String,
        default: "legacy",
    },
}
```

迁移过程：

1. 未改造页面保持 `legacy` 默认值。
2. 已改造页面显式传入 `modern`。
3. 全部页面迁移完成后，将默认值调整为 `modern`。
4. 完成全量回归后删除 `legacy` 分支。

如果现有组件难以安全增加 variant，应新增设计系统包装组件，不得强行覆盖公共默认样式。

## 8. 逐步实施与验收门禁

改造采用“单步实施 → 自动检查 → 视觉对比 → 用户验收 → 进入下一步”的方式。每一步必须独立交付；没有得到用户明确验收，不得提前实施下一步，也不得把下一步改动混入当前步骤。

### 第 0 步：建立改造基线

工作内容：

1. 确认工作区状态和已有改动。
2. 固定目标页面与尚未进入当前阶段页面的关键截图。
3. 记录桌面端、平板端和移动端基准。
4. 记录导航展开、收起、激活和滚动状态。
5. 运行 lint、兼容性和开发构建检查。
6. 整理公共组件引用关系。

验收标准：

- 当前项目能够正常构建。
- `/pvx/`、`/face/`、`/body/`、`/homeland/` 均可访问。
- 图鉴和信息类抽查页面均可访问。
- 基准截图和检查结果完整。
- 本步骤不改变任何页面视觉。

### 第 1 步：建立公共设计基础

工作内容：

1. 建立颜色、文字、圆角、阴影和间距令牌。
2. 建立 Tailwind 与 LESS 的职责边界。
3. 建立页面壳、Surface、区块标题等基础组件。
4. 建立 `legacy` / `modern` 渐进迁移机制。
5. 基础组件暂不接入业务页面。

验收标准：

- 公共设计组件可以独立渲染。
- Tailwind 构建和 LESS 编译正常。
- 未接入新组件的页面像素表现不变。
- 不新增无作用域的公共选择器覆盖。
- 自动检查全部通过。

### 第 2 步：改造 PVX 首页

工作内容：

1. 改造 `/pvx/` 首页布局，保留原有栏目和入口。
2. 使用浅灰背景、白色面板、轻量边框和阴影。
3. 统一分类标题和应用入口矩阵。
4. 优化背景插画、间距、Hover 和响应式布局。

验收标准：

- 所有入口完整且跳转正确。
- 1440、1280、768、390 宽度下布局正常。
- 无横向溢出、遮挡或卡片错位。
- Hover、点击和键盘焦点状态清晰。
- 导航及尚未进入当前阶段的页面不变。

### 第 3 步：改造捏脸与体型公共列表框架

工作内容：

1. 为公共搜索组件增加向后兼容的新版 variant。
2. 改造 Face/Body 共用工具栏。
3. 统一分类、搜索、筛选、发布和解析入口。
4. 建立新版列表卡、分页、加载和空状态。

验收标准：

- 默认 `legacy` 外观不变。
- 只有显式启用 `modern` 的页面出现新版样式。
- 搜索、筛选、分类切换和分页逻辑正常。
- 不产生重复请求或状态丢失。
- 考试、家具等使用公共搜索组件的页面不变。

### 第 4 步：完成捏脸列表页

工作内容：

1. 将 `/face/` 接入新版列表框架。
2. 调整卡片图片、标签、标题和作者信息。
3. 优化推荐列表、全部列表、加载更多和分页。
4. 保留付费、捏脸码、推荐和可新建状态。

验收标准：

- 全部角色分类和写实、写意等筛选正常。
- 搜索、分页和加载更多正常。
- 卡片详情与作者链接正常。
- 空数据和加载状态清晰。
- 体型及其他页面不发生视觉变化。

### 第 5 步：完成体型列表页

工作内容：

1. 将 `/body/` 接入新版列表框架。
2. 复用捏脸公共能力。
3. 仅保留图片比例、卡片高度和业务标签差异。

验收标准：

- 所有分类、筛选、搜索和分页正常。
- 图片不拉伸且裁切合理。
- 卡片信息层级与捏脸统一。
- 不复制一套与捏脸近似的冗余样式。
- 捏脸列表不出现回归。

### 第 6 步：改造捏脸详情页

工作内容：

1. 改造顶部信息、预览和购买下载区域。
2. 统一作者、标签、说明和附件列表。
3. 优化数据分析、推荐、投币和评论区块。
4. 保留捏脸码、支付、下载和管理行为。

验收标准：

- 免费、付费、已购买和未购买状态正确。
- 复制、单文件下载和批量下载正常。
- 支付和余额不足流程不变。
- 作者、推荐、投币和评论功能可用。
- 无权限、下架和加载失败状态正常。

### 第 7 步：改造体型详情页

工作内容：

1. 复用捏脸详情公共结构。
2. 接入体型数据分析和附件能力。
3. 保留体型特有的图片与数据展示差异。

验收标准：

- 购买、支付轮询、下载和附件列表正常。
- 数据分析解锁条件正确。
- 与捏脸详情保持统一视觉语言。
- 不复制捏脸详情的大量结构与样式。
- 捏脸详情不出现回归。

### 第 8 步：改造捏脸与体型解析页

工作内容：

1. 改造 `/face/facedata` 和 `/body/bodydata`。
2. 建立统一上传卡片和指南入口。
3. 将解析结果作为独立内容面板展示。
4. 移除依赖固定大 margin 的居中方式。

验收标准：

- 正确文件能够解析和预览。
- 错误文件有清晰提示且不导致页面崩溃。
- 未解析和已解析状态切换自然。
- 桌面和移动端没有异常空白或溢出。

### 第 9 步：改造家园首页与功能导航
