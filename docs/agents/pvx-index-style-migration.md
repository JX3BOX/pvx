# PVX 全站风格统一改造进度

## 1. 文档定位

本文只记录 PVX 模块改造范围、实施顺序、业务保护要求、验收状态和后续 TODO。

视觉风格、页面布局、设计令牌、基础组件外观、响应式规则和样式隔离规范统一维护在：

- `docs/design/pvx-visual-style-guide.md`

开始任何页面改造前必须先阅读该设计规范。本文不再重复维护视觉细节。

## 2. 改造边界

改造只调整表现层和必要的公共结构，不得改变：

- 接口地址、请求参数和返回数据语义；
- 登录、权限、作者和管理员判断；
- 购买、支付轮询、附件获取和下载流程；
- 路由名称、跳转行为、统计与埋点；
- 第三方业务组件自身的数据解析和国际化逻辑。

未进入当前阶段的页面不得提前受到样式影响。

## 6. 单模块实施流程

每个后续模块按以下顺序执行，并在用户验收后再进入下一模块：

1. 阅读路由、入口页面、子页面、业务组件、接口和现有样式。
2. 阅读视觉设计规范，对照已验收页面梳理可复用能力。
3. 列出必须保留的业务差异、接口行为和终端差异。
4. 建立模块专有根节点，确保样式不会泄漏到其他页面。
5. 先完成桌面端核心结构，再补齐平板与移动端。
6. 检查多语言长文本、空数据、加载、错误和权限状态。
7. 回归接口参数、购买、下载、跳转、统计等原有功能。
8. 运行自动检查，并按设计规范中的建议视口进行浏览器验收。
9. 用户验收通过后更新本 TODO；同类模块稳定后再抽取公共能力。

## 7. 业务验收标准

- 接口请求数量和参数符合原逻辑。
- 登录、作者、管理员和付费权限判断不变。
- 购买确认、支付轮询、余额不足、成功和失败提示正常。
- 下载入口、附件列表、单文件下载和批量下载正常。
- 路由跳转、外链、统计和评论等能力正常。
- 第三方组件的数据解析、上传和国际化行为不变。
- 小程序和 App 端未进入改造范围时不受影响。

视觉验收使用 `docs/design/pvx-visual-style-guide.md` 中的检查清单。

## 8. 代码质量与自动检查

- 公共 props 和 slots 语义清晰，没有为单页场景过度抽象。
- 没有重复包装第三方组件的数据解析或国际化。
- 没有业务无关的截图、临时文档或调试文件进入 Git。
- 新增公共能力后删除已被替代的重复代码。
- 未改造页面的旧代码不因当前任务被顺带重写。

固定自动检查：

```bash
npm run lint
npm run verify:legacy-js
npm run build:dev
```

## 9. 文档维护规则

- 本文只更新模块状态、实施顺序、业务风险和验收结论。
- 视觉规范变化只更新 `docs/design/pvx-visual-style-guide.md`。
- 新视觉模式先在业务页面落地并验收，确认稳定复用后再写入设计规范。
- 不保留逐日流水账和已完成步骤的冗长过程记录。
