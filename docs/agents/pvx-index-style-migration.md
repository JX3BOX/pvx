# PVX 全站风格统一改造规范

## 1. 文档定位

本文记录已经验收并可复用的视觉、响应式和代码基线，用于继续改造 PVX 其余业务模块。

首批改造与公共能力收敛已经完成并验收，历史实施步骤和逐日记录不再保留。后续改造应直接遵循本文，不得重新复制首批页面中的近似结构和样式。

改造只调整表现层和必要的公共结构，不得改变：

- 接口地址、请求参数和返回数据语义；
- 登录、权限、作者和管理员判断；
- 购买、支付轮询、附件获取和下载流程；
- 路由名称、跳转行为、统计与埋点；
- 第三方业务组件自身的数据解析和国际化逻辑。

## 3. 视觉与布局基线

### 3.1 视觉语言

- 页面背景使用浅灰蓝色，内容主体使用白色卡片。
- 卡片采用浅边框、克制阴影和明显圆角，避免高饱和大色块。
- 主文字使用深色 Slate 色系，说明文字降低对比度。
- 靛蓝或紫色作为主要强调色；黄色、玫红仅用于少量状态提示。
- 标题优先采用“图标块 + 主标题 + 副标题”的信息层级。
- Hover 动效以轻微上移、阴影增强和图标小幅缩放为主。
- 页面上下左右留白应使用统一容器规则，不能逐页凭视觉填写任意数值。

### 3.2 响应式原则

- 桌面端优先保证信息密度、稳定间距和完整操作入口。
- 平板端允许横向滚动承载固定宽度内容，但操作栏不能异常放大或挤压。
- 移动端使用单列结构；筛选、搜索和主要按钮必须保持可操作。
- 列表卡片不得被容器裁切，卡片间距在不同数量下必须一致。
- 页面主体应有合理最小高度，常规首屏不应直接露出页脚。
- 任何断点下都不得出现页面级非预期横向滚动。

建议检查视口：

| 类型 | 建议尺寸 |
| --- | --- |
| 大屏桌面 | `1920 × 1080` |
| 常规桌面 | `1440 × 900` |
| 小屏桌面 | `1280 × 800` |
| 平板 | `1024 × 700` |
| 移动端 | `390 × 844`、`414 × 896` |

## 4. 公共设计基础

### 4.1 设计令牌和 Mixins

公共色彩、圆角、阴影、间距、容器宽度和动效应优先复用：

- `src/assets/css/design-system/_tokens.less`
- `src/assets/css/design-system/_mixins.less`

同一视觉值在三个及以上已改造页面稳定复用时，才考虑提升为公共令牌。仅属于单一业务状态的颜色或尺寸保留在模块主题中，避免过度抽象。

### 4.2 基础视觉组件

已建立的基础组件：

| 组件 | 用途 |
| --- | --- |
| `PvxPageShell` | 页面背景、内容宽度、边距和最小高度 |
| `PvxSurface` | 白色卡片、边框、圆角、阴影和内边距 |
| `PvxSectionHeader` | 图标、标题、副标题和右侧操作区 |
| `PvxToolbar` | 搜索、筛选和主要操作布局 |
| `PvxActionButton` | 统一操作按钮外观和状态 |
| `PvxEmptyState` | 空结果、无数据和引导信息 |

新页面应先评估这些组件能否满足需求。只有存在稳定、重复且有明确语义的结构时才新增公共组件。

### 4.3 Face / Body 公共能力

捏脸与体型模块已收敛以下公共结构：

- `Tabs.vue`：列表筛选与搜索工具栏；
- `ListItem.vue`：列表卡片；
- `SingleNavigation.vue`：详情页返回和导航；
- `SingleHeader.vue`：详情页作者与基础信息；
- `SinglePaySection.vue`：购买、附件与下载操作区；
- `SingleCarousel.vue`：详情预览与全屏单图切换；
- `SingleRandomList.vue`：相关推荐；
- `ParsePage.vue`：Face / Body 解析页公共外壳；
- `pc-list-mixin.js`：列表请求、数量计算和筛选逻辑；
- `pc-single-mixin.js`：详情、购买、支付轮询、附件与下载公共逻辑。

Face / Body 页面只保留真实业务差异，例如接口方法、数据解析器、类型标识、文案键和个别模块操作。

第三方 `@jx3box/jx3box-facedat` 组件应直接使用。不得复制其内部国际化，也不得增加 `bodydatai18n` 一类重复包装；样式覆盖必须限定在当前模块根节点内。

## 5. Tailwind 与 LESS 约定

项目已全局引入 Tailwind CSS。两者按职责配合使用：

### Tailwind 适用范围

- 页面和组件的一次性布局；
- Flex、Grid、间距、对齐和响应式断点；
- 简单文字、尺寸、边框、背景和显示状态；
- 不需要跨组件共享的轻量交互样式。

### LESS 适用范围

- 设计令牌和公共 Mixins；
- 多状态、伪元素、渐变和复杂动画；
- 第三方组件的受控覆盖；
- 需要统一维护的模块主题和响应式规则。

### 禁止事项

- 不要为已有 Tailwind 能力重复编写一套工具类。
- 不要把长串 Tailwind 类机械复制到多个页面；稳定复用后应抽成组件。
- 不要因为抽取公共组件而把业务差异塞进大量布尔参数。
- 不要把单一页面的偶然尺寸提升为全局令牌。

## 6. 样式隔离与渐进迁移

未改造页面继续保持原有样式。现代样式只能由明确根节点开启，例如：

- `.p-pvx-index`
- `.p-pvx-face-*`
- `.p-pvx-body-*`
- `.p-pvx-homeland-*`

禁止新增无作用域的通用选择器覆盖，例如：

- `body`
- `.m-main`
- `.u-title`
- `.u-icon`
- `.u-btn`
- `.el-*`

如需覆盖第三方组件，至少应从页面或公共组件的专有根类开始。公共组件应由调用方通过 `modern`、`variant` 或明确 class 主动启用新版外观，不能让全站实例自动变化。

公共样式抽取顺序：

1. 先确认结构在多个已改造页面中稳定重复；
2. 抽取布局与视觉骨架；
3. 用少量语义明确的 props、slots 暴露差异；
4. 保留接口调用、权限和业务状态在模块层；
5. 删除已被公共能力替代的重复代码；
6. 对已验收页面和未改造页面同时做回归。

## 8. 单模块实施流程

每个后续模块按以下顺序执行，并在用户验收后再进入下一模块：

1. 阅读路由、入口页面、子页面、业务组件、接口和现有样式。
2. 对照本规范与已验收页面，列出可复用能力和必须保留的业务差异。
3. 建立模块专有根节点，确保样式不会泄漏到其他页面。
4. 先完成桌面端核心结构，再补齐平板与移动端。
5. 检查多语言长文本、空数据、加载、错误和权限状态。
6. 回归接口参数、购买、下载、跳转、统计等原有功能。
7. 运行自动检查，并按建议视口进行浏览器验收。
8. 用户验收通过后更新本 TODO；同类模块稳定后再抽取公共能力。

## 9. 验收标准

### 9.1 视觉与交互

- 页面背景、容器边距、卡片、圆角、阴影和文字层级符合基线。
- 桌面、平板和移动端没有裁切、重叠、异常空白或页面级横向溢出。
- 长标题和中、英、越南文不会遮挡图标或操作按钮。
- 空结果时不渲染无内容的分类外框。
- 首屏高度合理，常规视口不直接露出页脚。
- Hover、点击、筛选、搜索、分页、弹层和预览行为正常。

### 9.2 业务回归

- 接口请求数量和参数符合原逻辑。
- 登录、作者、管理员和付费权限判断不变。
- 购买确认、支付轮询、余额不足、成功和失败提示正常。
- 下载入口、附件列表、单文件下载和批量下载正常。
- 路由跳转、外链、统计和评论等能力正常。

### 9.3 代码质量

- 公共 props 和 slots 语义清晰，没有为单页场景过度抽象。
- 没有新增裸全局选择器或不受控的 Element Plus 覆盖。
- 没有重复包装第三方组件的数据解析或国际化。
- 没有业务无关的截图、临时文档或调试文件进入 Git。
- 自动检查通过：

```bash
npm run lint
npm run verify:legacy-js
npm run build:dev
```
