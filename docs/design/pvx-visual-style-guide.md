# PVX 视觉风格与界面设计规范

## 1. 文档定位

本文是 PVX 项目已经落地并通过验收的视觉设计基线。后续页面改造必须优先参考本文以及现有设计令牌、基础视觉组件，避免重新复制近似样式。

本文只描述视觉、布局、响应式、组件外观和样式工程规则。模块进度、业务保护范围和实施顺序记录在：

- `docs/agents/pvx-index-style-migration.md`

当前视觉基线来源于已经验收的 PVX 首页、捏脸数据、体型数据和家园蓝图模块。

## 2. 整体视觉语言

### 2.1 页面层级

- 页面背景使用浅灰蓝色，当前基础色为 `#f6f8fa`。
- 内容主体使用白色卡片，与页面背景形成柔和层级。
- 卡片采用浅色边框、克制阴影和明显圆角，避免大面积高饱和背景。
- 主要内容通过卡片分组，不依赖粗重分割线。
- 页面上下左右留白必须遵循统一容器规则，不能逐页填写任意边距。
- 页面主体应有合理最小高度，常规首屏不应直接露出页脚。

### 2.2 色彩

- 主文字使用深色 Slate 色系。
- 次要文字和说明文字降低对比度，但必须保持可读性。
- 靛蓝或紫色作为主要强调色，用于激活、主按钮和重点信息。
- 黄色、玫红等色彩只用于少量状态、提醒或业务语义。
- 免费、付费、收藏、服务器状态等业务色保留在模块层，不强制提升为全局色。
- 禁止为了“更醒目”在普通内容区使用大面积高饱和底色。

### 2.3 圆角、边框和阴影

- 页面主卡片使用较大圆角。
- 卡片内部的信息块、筛选器和按钮使用中小圆角。
- 边框使用低对比度 Slate 灰色，并优先使用带透明度的细边框。
- 默认阴影应克制，只承担层级表达。
- Hover 可通过轻微上移、边框强调和阴影增强反馈，不使用大幅位移。

### 2.4 文字层级

- 页面主标题应清晰但不过度巨大。
- 模块标题建议使用“图标块 + 主标题 + 副标题”结构。
- 卡片标题、辅助说明、元信息和状态文字应形成明确字号及字重层级。
- 长标题必须截断或合理换行，不能挤压图标、标签和操作按钮。
- 中文、英文和越南文均应检查，不能只按中文长度设计。

## 3. 页面容器与间距

统一页面外壳由 `PvxPageShell` 承载：

- 页面最大内容宽度：`1440px`；
- 常规桌面水平内边距：`32px`；
- 中等屏幕水平内边距：`24px`；
- 移动端水平内边距：`16px`；
- 常规桌面垂直内边距：`32px`；
- 移动端垂直内边距：`20px`；
- 默认应提供覆盖首屏的最小高度。

页面内优先使用稳定的间距层级：

- 紧凑控件间距：`8px`；
- 普通控件间距：`12px`；
- 卡片内部区块间距：`16px`；
- 页面区块间距：`20px` 或 `24px`；
- 大型内容段落间距：`32px`。

相同层级的卡片必须保持一致间距。列表数量变化时，不能通过 `space-between` 把卡片间距拉伸为不同数值。

## 4. 卡片与常用界面结构

### 4.1 页面主卡片

主卡片用于页面级内容分组：

- 白色背景；
- 浅色边框；
- 大圆角；
- 轻阴影；
- 桌面端使用较宽松内边距，移动端自动收紧。

优先使用 `PvxSurface`，不要在每个业务页面重新声明同一套背景、边框、圆角和阴影。

### 4.2 页面和区块标题

优先使用 `PvxSectionHeader`：

- 可选图标块；
- 主标题；
- 可选副标题；
- 右侧操作区；
- 移动端自动收紧图标和字号。

业务区块内部的小标题可以使用更紧凑的标题样式，但必须放在模块专有根节点内，不得新增裸 `.u-title` 全局规则。

### 4.3 工具栏

搜索、筛选、分类和主要操作优先放入 `PvxToolbar`：

- 桌面端保持一行并合理分配剩余宽度；
- 平板端不得因为空间变化而异常放大按钮；
- 移动端允许重排，但搜索和筛选应保持相邻且清晰；
- 主要操作按钮不能遮挡文字或被装饰图片覆盖。

### 4.4 操作按钮

- 主操作使用明确强调色。
- 次操作使用白底、浅边框或轻量文字按钮。
- 图标按钮必须有明确可点击区域和可访问名称。
- 禁用状态降低对比度，同时取消误导性的 Hover 动效。
- 长语言环境下允许合理缩短文案、换行或使用图标，但必须保留可理解性。

### 4.5 空状态

- 无数据时不渲染空白分类卡或无内容外框。
- 页面只保留一处明确空状态说明。
- 空状态应使用 `PvxEmptyState` 或同等语义结构。
- 空状态不应占据不必要的大面积高度。

## 5. 基础视觉组件

现有组件位于 `src/components/design`：

| 组件 | 用途 |
| --- | --- |
| `PvxPageShell` | 页面背景、最大宽度、统一边距和最小高度 |
| `PvxSurface` | 白色卡片、边框、圆角、阴影和内边距 |
| `PvxSectionHeader` | 图标、标题、副标题和右侧操作区 |
| `PvxToolbar` | 搜索、筛选和主要操作布局 |
| `PvxActionButton` | 统一操作按钮外观和状态 |
| `PvxEmptyState` | 空结果、无数据和引导信息 |

使用原则：

1. 新页面先评估现有组件是否满足需求。
2. 只有结构在多个页面中稳定重复时才新增公共组件。
3. 公共组件使用少量语义明确的 props 和 slots 表达差异。
4. 不把单页差异转化为大量布尔参数。
5. 业务接口、权限和业务状态继续保留在模块层。

## 6. 设计令牌与 Mixins

公共色彩、圆角、阴影、间距、容器宽度和动效优先复用：

- `src/assets/css/design-system/_tokens.less`
- `src/assets/css/design-system/_mixins.less`

抽取规则：

- 同一视觉值在三个及以上已改造页面稳定复用时，才考虑提升为公共令牌。
- 仅属于单一业务状态的颜色或尺寸保留在模块主题中。
- 不把单一页面的偶然尺寸提升为全局令牌。
- 新增令牌前先确认现有令牌和 Tailwind 配置无法满足需求。

## 7. Tailwind 与 LESS 使用规范

项目已全局引入 Tailwind CSS，两者按职责配合使用。

### 7.1 Tailwind 负责

- 页面和组件的一次性布局；
- Flex、Grid、间距、对齐和响应式断点；
- 简单文字、尺寸、边框、背景和显示状态；
- 不需要跨组件共享的轻量交互样式。

### 7.2 LESS 负责

- 设计令牌和公共 Mixins；
- 多状态、伪元素、渐变和复杂动画；
- 第三方组件的受控覆盖；
- 需要统一维护的模块主题和复杂响应式规则。

### 7.3 禁止事项

- 不为已有 Tailwind 能力重复编写工具类。
- 不把长串 Tailwind 类机械复制到多个页面。
- 不用 `!important` 掩盖错误的样式层级，第三方兼容场景除外且必须限定作用域。
- 不在模板中堆积难以理解的动态字符串来代替语义 class。

## 8. 响应式规范

### 8.1 桌面端

- 优先保证信息密度、稳定间距和完整操作入口。
- 网格列数根据可用宽度调整，不允许最后一个卡片被裁切。
- 大屏可以增加列数，但不随意拉伸卡片间距。

### 8.2 平板端

- 操作栏保持正常按钮尺寸，不能自动放大成大面积块状按钮。
- 固定宽度内容确有必要时可以局部横向滚动。
- 横向滚动只能存在于明确内容容器，不能形成页面级横向滚动。

### 8.3 移动端

- 页面主体使用单列布局。
- 列表内容优先自然换行；除明确横向浏览场景外，不使用横向滚动。
- 筛选和搜索应保持同一操作层级。
- 卡片图片可根据业务改为占满内容宽度。
- 弹层宽度和定位应相对视口居中，并保留安全边距。

### 8.4 建议检查视口

| 类型 | 建议尺寸 |
| --- | --- |
| 大屏桌面 | `1920 × 1080` |
| 常规桌面 | `1440 × 900` |
| 小屏桌面 | `1280 × 800` |
| 平板 | `1024 × 700` |
| 移动端 | `390 × 844`、`414 × 896` |

所有视口必须检查：

- 页面级横向溢出；
- 内容裁切和重叠；
- 卡片间距；
- 长文本；
- 首屏与页脚位置；
- 筛选、搜索、弹层和主要操作可用性。

## 9. 交互与状态

- Hover 动效以轻微上移、图标小幅缩放和阴影增强为主。
- 动效持续时间应短且克制，避免影响信息读取。
- 激活状态主要使用强调色、浅色底和边框表达。
- 加载、成功、失败、禁用和空状态必须视觉可区分。
- 切换控件应保留稳定尺寸，避免选中后引发布局跳动。
- 图片预览一次只展示一张大图；支持的页面应提供上一张、下一张和移动端滑动。

## 10. 多语言与可访问性

- 所有新增业务文案使用现有 i18n 体系，避免继续增加中文硬编码。
- 英文和越南文必须检查按钮、标签、标题和卡片元信息。
- 长文本可通过缩短翻译、合理换行、截断和 Tooltip 处理。
- 可点击的非原生元素必须具备明确交互语义；优先使用 `button`、`a` 等原生语义标签。
- 图标按钮需要 `title`、`aria-label` 或可见文字。
- 外链使用明确跳转方式，并保留 `rel="noopener noreferrer"`。

## 11. 样式隔离与渐进迁移

未改造页面继续保持原有视觉。现代样式只能由明确根节点开启，例如：

- `.p-pvx-index`
- `.p-pvx-face-*`
- `.p-pvx-body-*`
- `.p-pvx-homeland-*`
- 后续模块对应的专有 `.p-pvx-*` 根节点。

禁止新增无作用域的通用选择器覆盖：

- `body`
- `.m-main`
- `.u-title`
- `.u-icon`
- `.u-btn`
- `.el-*`

如需覆盖 Element Plus 或第三方组件，至少应从页面或公共组件的专有根类开始。

公共组件应由调用方通过 `modern`、`variant` 或明确 class 主动启用新版外观，不能让全站旧实例自动变化。

第三方业务组件应直接使用其数据解析和国际化能力，不复制内部实现。外观覆盖必须限定在当前模块根节点内。

## 12. 视觉验收清单

- [ ] 页面背景、容器边距和最小高度符合统一外壳。
- [ ] 卡片背景、边框、圆角和阴影层级一致。
- [ ] 标题、说明、元信息和状态文字层级清晰。
- [ ] 桌面、平板和移动端没有裁切、重叠或页面级横向溢出。
- [ ] 相同层级卡片间距始终一致。
- [ ] 长标题和中、英、越南文不会遮挡图标或操作按钮。
- [ ] 空结果时不渲染无内容的分类外框。
- [ ] 常规首屏不直接露出页脚。
- [ ] Hover、点击、筛选、搜索、分页、弹层和预览行为正常。
- [ ] 未新增裸全局选择器或不受控的第三方组件覆盖。
- [ ] 未改造页面没有视觉回归。

## 13. 规范维护

- 新视觉模式先在业务页面落地并完成验收。
- 在多个模块中确认稳定复用后，再补充到本文和公共组件。
- 业务专属视觉不写成强制全局规范。
- 修改本文时同步检查设计令牌和基础视觉组件是否需要调整。
- 迁移进度变化只更新迁移文档，不在本文记录逐日实施日志。
